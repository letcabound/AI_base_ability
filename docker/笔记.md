## 1.1 相同宿主机的Docker服务的端口映射和网络路由
  服务器上用docker启动了一个服务，启动的时候，配置文件设置了端口映射。宿主机在接收外部机器B的请求的时候，Docker的 iptables规则，会将数据包的
目标地址修改为容器ip(DNT)，然后数据包会进入容器； 数据在容器内经过处理后，容器回复给宿主机器且源地址为容器ip，Docker会再次把源地址还原为宿主机ip(SNT)，
外部机器B接收到的响应数据后发现 TCP连接跟踪表里记录的"请求-响应"五元组完全对称，握手成功。

## 1.2 Docker容器 vs 物理机
  a. 从业务使用来看，Docker容器和正式服务器(物理机/虚拟机)差不多，技术上来看 二者的核心区别在于 '内核'。相比于独立服务器，Docker容器同样有独立的文件
系统、进程空间、网络ip和端口、独立的root用户。
  b. 从应用软件的角度来看，你把以恶程序扔进容器里，和在物理机上安装运行几乎一样。
  c. 二者最大的区别：Docker容器没有自己的内核，和宿主机共用同一个Linux内核。物理机/虚拟机 的资源是硬隔离的，拥有独立的内核和内存：内核参数可以修改且内存资源完全占据硬隔离。而DOcker容器的资源是
软隔离的：容器内部无法修改内核参数，内存资源也只是设置的上限，空闲内存还是归宿主机和其他容器共用。相当于一个被限制了内存使用上限的普通进程。

## 1.3 Docker镜像和容器
  Docker通过镜像新启动容器就相当于新建了一个独立的服务器和应用服务。镜像，封装了【应用代码，环境变量，启动命令以及操作系统版本】的只读模板。 容器， 
在启动的时候会默认无限制占用宿主机的资源(假设宿主机部署了三个微服务，都没有加资源限制。其中一个发生资源泄露，会直接把宿主机的内存吃光，导致宿主机卡死，
另外两个无辜的服务也会跟着宕机，这就是著名的"吵闹的邻居"问题。)，除非每次启动的时候手动临时添加参数【比如：docker run --memory=512m --cpus=1 ...】，
或者永久性的将资源占用参数写在 docker-compose.yaml 文件里。

## 1.4 Docker容器的命令
  docker compose up -d --build：根据项目代码新建一个镜像，如果修改了 业务代码 或者 Dockerfile文件，必须重新 build。
  docker compose up -d：根据镜像新建一个容器，如果修改了环境变量且配置文件没有挂载，则需要 up。(环境变量是新建容器的时候注入的)
  docker compose restart：关闭当前容器，再启动。如果修改了挂载文件的内容，可使用此命令。(宿主机的挂载文件是实时穿透容器的。)
  总结：build会从头到尾根据代码对读取整个项目文件并下载依赖 打包为镜像。新建容器只是将已经包含所有依赖的镜像复制一份然后隔离开，直接启动里面的主程
序。容器新建的时候，会一次性读取环境变量配置。后续容器运行后，每次重启都会重新读取宿主机的挂载文件。(讲道理挂载文件 宿主机和容器是同一份，但是容器内
的会将文件加载到内存，内存的老代码会一直生效，所以才需要重启。)


## 2.1 Docker -> Docker Compose
  真实的业务系统，通常是多个服务的协同，常见的如 "Web应用 + MySQL + Redis" 的标准项目。 如果只是用Docker启动，会出现问题：1)每个服务依赖都要
写一堆 -v(挂载) -e(环境变量) -p(端口)，Compose中通过极简的KV配置来规范化；2)不同依赖服务之间的存在启动顺序问题，Compose的配置文件通过字段
depends_on 来指定服务的依赖关系，决定容器的启动顺序。3)为了让同一个宿主机的容器A访问到容器B(DNT,SNT问题)，需要新建一个网络然后每个容器都加入到
该网络。compose则会自动为项目创建独立的专属网络，直接用服务名就能互相访问，不需要知道IP。
  总之，Docker Compose解决了Docker在微服务架构下"管理复杂，配置分散，依赖混乱"的最大短板，让容器技术真正具备了大规模协同落地的能力。


## 3.1 Docker / Docker Compose 查看日志命令
  容器启动日志和应用服务运行日志都是：
    docker (compose) logs -f 服务名
    docker (compose) logs 服务名 --tail=100


## 实操案例：Mem0 / Ollama
问题1：服务器本地启动的docker服务，在服务器本地访问会被reset。
    现象：
        Docker部署的服务，外部机器B访问正常，但是宿主机本地访问则失败。
    报错解释：
        Docker经典的 hairpin NAT(发夹 NAT)问题。
    永久解决方案
      让 mem0 和 Ollama 走同一个 Docker 网络，用容器名直接通信，彻底绕过 iptables NAT：
      mem0容器 → ollama:11434 → Ollama容器
          ↑ 同一 Docker bridge 网络，直接路由，不经过 iptables DNAT ↑
      例如：docker-compose.yaml 配置文件里 mem0 用 host: "postgres" 容器名访问数据库的。
